API概述

企业微信通知服务提供了简单易用的HTTP API，用于向企业微信应用发送消息通知。通过配置您的企业微信应用信息，系统会生成一个唯一的API地址，您可以通过该地址发送消息。

使用前，您需要先在首页创建配置并获取唯一的API地址。

认证与安全

通知发送 API 支持两级安全策略，通知密钥默认关闭，可按需在管理后台「配置详情 → 通知密钥」中启用：

模式 请求头 说明
默认（无鉴权） 无需额外头 仅凭 /api/notify/{code} 即可发送。适合内网/受信环境快速接入。
通知密钥（可选） X-Notify-Key 启用后必须携带正确的 notify_key，错误或缺失返回 401。密钥在配置详情页生成，仅显示一次，可随时轮换/撤销。
HMAC 签名（可选增强） X-Notify-Key +
X-Notify-Timestamp +
X-Notify-Nonce +
X-Notify-Signature		 在通知密钥基础上增加签名校验，拒绝重放与篡改。详见下方 HMAC 签名模式。
通知密钥与回调地址使用的 Code 相互分离，互不影响。规则（Rule）API 与其所属配置共用同一把 notify_key。
管理后台登录凭证改由 HttpOnly Cookie 携带，不再出现在 URL 或浏览器脚本可读的存储中。通知密钥仅在生成时显示一次，请妥善保存；遗失或泄漏后可随时轮换。

HMAC 签名模式（可选增强）

启用通知密钥后，可额外提供三个签名头启用 HMAC-SHA256 校验，防止请求被重放或篡改。一旦提供任意一个签名头，三个头必须同时提供且校验通过。

签名头说明
请求头 说明
X-Notify-Timestamp Unix 秒级时间戳，与服务端相差不得超过 ±300 秒，否则返回 401。
X-Notify-Nonce 随机串（建议 16+ 位），服务端在 10 分钟内拒绝相同 nonce，防止重放。
X-Notify-Signature HMAC-SHA256 签名的十六进制串（小写）。
待签字符串（用 \n 连接 5 段）
{HTTP方法}
{请求路径，如 /api/notify/your-code}
{时间戳，与 X-Notify-Timestamp 一致}
{nonce，与 X-Notify-Nonce 一致}
{请求体的 SHA-256 十六进制，即 sha256(rawBody)}

注意：第 5 段是对客户端实际发送的原始请求体字节做 SHA-256，不要对 JSON 重新序列化（空白/换行不同会导致签名失败）。

签名计算（伪代码）
secret        = hex2bytes(notify_key)        # notify_key 的十六进制字节作为 HMAC 密钥
bodyHash      = sha256(rawBody).hex()        # 请求体哈希
message       = [method, path, timestamp, nonce, bodyHash].join("\n")
signature     = hmac_sha256(secret, message).hex()
完整 curl 范例（含通知密钥 + HMAC 四头）
#!/bin/bash
CODE="your-code-here"
KEY="你的 notify_key"
BODY='{"title":"服务器告警","content":"CPU使用率超过90%"}'

TS=$(date +%s)
NONCE=$(head -c 16 /dev/urandom | xxd -p)
BODY_HASH=$(printf '%s' "$BODY" | sha256sum | awk '{print $1}')
MSG=$(printf 'POST\n/api/notify/'"$CODE"'\n'"$TS"'\n'"$NONCE"'\n'"$BODY_HASH")
SIG=$(printf '%s' "$MSG" | openssl dgst -sha256 -mac HMAC -macopt hexkey:"$KEY" | awk '{print $NF}')

curl -X POST "https://your-server.com/api/notify/$CODE" \
  -H "Content-Type: application/json" \
  -H "X-Notify-Key: $KEY" \
  -H "X-Notify-Timestamp: $TS" \
  -H "X-Notify-Nonce: $NONCE" \
  -H "X-Notify-Signature: $SIG" \
  -d "$BODY"

未启用通知密钥时，可省略以上全部四个 X-Notify-* 头，直接发送即可。

HMAC 签名内容包含请求路径（如 /api/notify/your-code）。若修改了规则的 api_code，调用方必须按新路径重新计算签名，否则旧路径签名会失效（旧地址本身也会立即 404）。规则仍继承所属应用的通知密钥，改号不改变密钥。

反向代理注意事项
HMAC 签名依赖服务端收到的原始请求体字节。若服务部署在 Nginx/OpenResty 等反代之后，且反代对请求体做了重编码（如改用 chunked transfer、重新序列化、追加/裁剪空白），会导致服务端 sha256(rawBody) 与客户端签名时的 body 哈希不一致，返回 401「签名校验失败」。排查方法：直连容器端口（绕过反代）若签名通过，即为反代改写 body 所致。解决：配置反代透传原始请求体（Nginx 用 proxy_pass + proxy_set_header，避免 client_body_buffer 重写；确保 proxy_request_buffering off 或保持原始 Content-Length）。通知密钥（仅 X-Notify-Key，不带签名）不受此影响。

发送消息API

请求地址

POST /api/notify/{your_code}

其中 {your_code} 是您创建配置时获得的唯一Code

请求头

Content-Type: application/json X-Notify-Key: <notify_key> # 启用通知密钥后必填，未启用可省略

未启用通知密钥时，仅需 Content-Type。如需 HMAC 签名增强，另见 HMAC 签名模式。

请求参数

参数名 类型 必填 说明
msgType String 否 消息类型，默认: text。可选值: text, markdown, image, file, textcard, news
title String 否 消息标题。文本/文本卡片/图文消息必填。
content String 否 消息内容。文本/Markdown/文本卡片消息必填。
mediaId String 否 媒体文件ID。图片/文件消息必填，需要通过素材上传接口获取。
url String 否 点击消息后跳转的链接。文本卡片/图文消息必填。
btntxt String 否 按钮文字。文本卡片消息可选，默认为"详情"。
articles Array 否 图文消息数组。图文消息必填，每项包含title,description,url,picurl。
safe Int 否 是否保密消息，0否(默认)，1是。图片/文件/文本卡片/图文消息支持。

请求示例

# 未启用通知密钥 —— 最简调用
curl -X POST "https://your-server.example/api/notify/your-code-here" \
-H "Content-Type: application/json" \
-d '{
  "title": "服务器告警",
  "content": "CPU使用率超过90%，请及时处理！\n\n**详细信息**：\n- 服务器：web-server-01\n- 时间：2023-09-15 14:30:45\n- 当前负载：95%"
}'

启用通知密钥后，请在上方追加 -H "X-Notify-Key: 你的密钥"；完整四头签名范例见 HMAC 签名模式。

接收规则 API

管理规则

接口 方法 说明
/api/configurations GET 获取可选择的基础配置 Code 列表，供规则页下拉选择
/api/configuration/{code}/rules GET 获取某个基础配置下的规则列表
/api/configuration/{code}/rules POST 创建规则并生成独立通知 API
/api/rules/{id} PUT 修改规则名称和接收范围
/api/rules/{id}/regenerate POST 重新生成规则 API，旧地址立即失效；旧编号会被保留，不能分配给其他规则
/api/rule-api-codes/availability GET 查询自定义编号是否可用（预检，不代替保存校验）
/api/rules/{id} DELETE 删除规则；编号会被保留，不能被新规则接管

配置列表响应

{
  "configurations": [
    {
      "code": "abc123",
      "agentid": 1000002,
      "description": "生产告警",
      "completed": true,
      "created_at": "2026-06-26 12:00:00"
    }
  ]
}

规则配置页仅自动选择已完成的配置；未完成配置会显示在列表中，但需要先回到首页补全应用信息和接收人。

规则请求体

{
  "name": "运维值班",
  "api_code": "ops-oncall",
  "is_all": false,
  "touser": ["zhangsan", "lisi"],
  "toparty": "2|3",
  "totag": "9",
  "estimated_count": 2
}

当 is_all 为 true 时，系统会按企业微信官方规则发送 touser="@all"，并忽略 touser/toparty/totag。不创建规则时，原有 /api/notify/{code} 仍按基础配置发送。

api_code 为可选的自定义通知编号（3～64 位小写字母、数字、-、_，首尾须为字母或数字）。创建时省略或留空则自动生成 UUID；编辑时省略表示不修改编号，显式空值会返回 400 RULE_API_CODE_INVALID。修改编号后旧地址 /api/notify/{旧编号} 立即失效，调用方需改用新地址。编号是路由标识，不是凭证——请勿当作密码使用。

编辑规则改号成功时，响应额外包含 "api_code_changed": true。编号在规则、应用 Code 与退役编号构成的全局命名空间内唯一（大小写不敏感）；被占用或已被保留时返回 409 RULE_API_CODE_CONFLICT（details.conflict_scope 为 rule/configuration/retired，不带 version，不会与版本冲突混淆）。

自定义通知 API

curl -X POST "https://your-server.example/api/notify/rule-api-code" \
-H "Content-Type: application/json" \
-d '{
  "title": "告警通知",
  "content": "磁盘使用率超过 90%"
}'

规则 API 与基础通知 API 的请求体一致。短时间内完全相同的消息会返回缓存结果并带有 response.cached=true；如需强制发送，可传 "force": true。

缓存与限频

  • access_token 继续按企业微信返回的 expires_in 缓存，并提前 5 分钟过期。
  • 规则/配置解析默认缓存 60 秒，成员列表默认缓存 5 分钟，成员刷新接口可用 ?refresh=1 强制重拉。
  • 显式成员发送会做本地保护：每成员每分钟 30 次、每小时 1000 次；触发后返回 HTTP 429。
  • 企业微信返回 45009 等限频错误时，当前应用会进入短暂退避保护，后续请求返回 HTTP 429。

应用管理 API（多应用）

一个应用 = configurations 的一行（企业 + AgentID + 凭证 + 默认接收人）。同一企业可有多个不同 AgentID 的应用，应用 Code 互不串用。

应用总览与详情

  • GET /api/configurations —— 按企业返回应用列表，每行含 lifecycle_status（draft/active/paused）、capabilities、warnings、version、recipient_count、rule_count。不返回 CorpSecret / AESKey / 通知密钥哈希。
  • GET /api/configuration/:code —— 应用详情，复用统一序列化；新增 corpsecret_configured/callback_token_configured/encoding_aes_key_configured（仅表示是否配置，不返回明文）。

三层发送开关（优先级从高到低）

配置 Code：app_enabled AND code_send_enabled AND notify_auth_ok
规则 API ： app_enabled AND rule.enabled       AND notify_auth_ok
  • 应用仍是草稿：409 APP_NOT_COMPLETED。
  • 应用总开关关闭：403 APP_DISABLED（配置 Code 与全部规则 API 均被拒）。
  • 配置 Code 入口关闭：403 DIRECT_SEND_DISABLED；规则关闭：403 RULE_DISABLED。
  • 暂停只影响发送，编辑/规则/安全设置仍可使用。

应用聚合写操作（需 If-Match 版本）

列表/详情返回 version；写请求携带 If-Match: <version>。缺版本 428 APP_VERSION_REQUIRED；版本不匹配 409 APP_VERSION_CONFLICT（响应含最新 version）。成功响应含 version（或规则的 app_version），前端采用服务端值，不本地 +1。

  • PUT /api/configuration/:code —— 局部更新（touched-field），空 CorpSecret/Token/AESKey 表示不变。corpid 不可改（400 CORPID_IMMUTABLE）；改 AgentID 按 (corpid, agentid) 判重（409 APP_IDENTITY_CONFLICT）。
  • DELETE /api/configuration/:code —— 事务级联删除应用及全部规则，返回 { configurations_deleted, rules_deleted }。
  • PUT /api/configuration/:code/app-enabled —— 发送启用/暂停切换（严格布尔，草稿拒绝）。暂停只阻止发送，不影响编辑、规则与安全设置管理。
  • PUT /api/configuration/:code/code-send —— 应用 Code 直接发送开关（不影响规则）。
  • POST/DELETE /api/configuration/:code/notify-key —— 启用/轮换/撤销通知密钥（轮换返回一次性明文 notify_key）。
  • POST /api/configuration/:code/rules、PUT/DELETE /api/rules/:id、POST /api/rules/:id/regenerate、PUT /api/rules/:id/enabled —— 规则增删改/重生成/启停，与应用版本原子提交，成功返回 app_version。
  • POST /api/generate-callback（携带 draft_code 更新草稿时）与 POST /api/complete-config —— 草稿更新/完成同样需 version（请求体字段，区分缺失 428 / 非法 400 / 冲突 409）；新建草稿返回初始 version=1。

创建流程（草稿隔离）

  • POST /api/generate-callback —— 生成草稿与回调 URL。同企业已有草稿：409 APP_DRAFT_EXISTS（details.existing_code）。
  • POST /api/complete-config —— 完成草稿。已完成：409 APP_ALREADY_COMPLETED；身份冲突：409 APP_IDENTITY_CONFLICT。
  • 草稿不会被当成可发送应用；草稿发送返回 409 APP_NOT_COMPLETED。

集成示例

下方示例均按未启用通知密钥的最简形式给出。启用通知密钥后，请在每个请求追加 X-Notify-Key 头；完整 HMAC 四头签名范例见 HMAC 签名模式。

Shell脚本示例

#!/bin/bash

# 监控服务器CPU使用率并发送告警
CPU_USAGE=$(top -bn1 | grep "Cpu(s)" | awk '{print $2 + $4}')
THRESHOLD=90

if (( $(echo "$CPU_USAGE > $THRESHOLD" | bc -l) )); then
  curl -X POST "https://your-server.example/api/notify/your-code-here" \
  -H "Content-Type: application/json" \
  -d "{
    \"title\": \"CPU告警\",
    \"content\": \"服务器CPU使用率: ${CPU_USAGE}%，超过阈值${THRESHOLD}%\"
  }"
fi

Python示例

import requests
import json

def send_notification(title, content):
    url = "https://your-server.example/api/notify/your-code-here"
    headers = {"Content-Type": "application/json"}
    data = {
        "title": title,
        "content": content
    }
    
    response = requests.post(url, headers=headers, data=json.dumps(data))
    return response.json()

# 使用示例
result = send_notification(
    "数据库备份完成", 
    "数据库备份已完成\n- 数据库：user_db\n- 备份大小：1.2GB\n- 备份时间：2023-09-15 02:00:00"
)
print(result)

Node.js示例

const axios = require('axios');

async function sendNotification(title, content) {
  try {
    const response = await axios.post('https://your-server.example/api/notify/your-code-here', {
      title,
      content
    }, {
      headers: {
        'Content-Type': 'application/json'
      }
    });
    
    return response.data;
  } catch (error) {
    console.error('发送通知失败:', error.response ? error.response.data : error.message);
    throw error;
  }
}

// 使用示例
sendNotification('部署完成', '项目已成功部署到生产环境\n\n版本: v1.2.3\n部署时间: 2023-09-15 15:30:00')
  .then(result => console.log('通知发送成功:', result))
  .catch(err => console.error('通知发送失败:', err));

常见问题

Q: 如何获取配置Code？

A: 在首页填写您的企业微信应用信息，验证并选择接收成员后，点击"生成通知API"按钮即可获取配置Code。

Q: 消息发送失败怎么办？

A: 请检查以下几点：

  • 配置Code是否正确
  • 请求参数格式是否正确
  • 企业微信应用的配置是否有效
  • 网络连接是否正常

Q: 是否支持发送图片或文件？

A: 目前仅支持发送文本消息，不支持图片或文件附件。

Q: 消息内容有长度限制吗？

A: 根据企业微信API的限制，消息内容不应超过2048个字符。
